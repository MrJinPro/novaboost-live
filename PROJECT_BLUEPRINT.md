# NovaBoost Live Blueprint

## Что это за проект

NovaBoost Live - это платформа для стримеров и зрителей вокруг TikTok LIVE.

Это не просто каталог стримов и не просто сервис уведомлений. Платформа должна:

- отслеживать эфиры стримеров в реальном времени
- автоматически подключать backend-трекинг к стримам зарегистрированных стримеров
- собирать live-события и активность зрителей
- распределять внешний и внутренний трафик между стримерами
- давать зрителям участие, очки, уровни, задания и награды
- давать стримерам инструменты роста, буста, подписок, контента и продвижения
- работать как web-продукт и как Telegram-экосистема

Главная идея продукта:

> зритель помогает стримеру расти, а платформа превращает это в живую экосистему участия, контента, бустов и удержания

## Продуктовые принципы

1. Платформа должна жить между эфирами, а не только во время эфира.
2. Страница стримера - это не просто карточка, а собственное пространство внутри продукта.
3. Зритель должен быть участником системы, а не пассивным получателем уведомлений.
4. Backend - это ядро продукта. Именно backend определяет жизнь системы: отслеживание, события, очки, сигналы, бусты, алгоритмы и уведомления.
5. Монетизация подключается позже, но архитектура сразу должна поддерживать платные бусты, подписки и премиум-оформление.
6. Все интеграции с TikTok должны строиться вокруг ручного ввода username пользователем. Никаких дефолтов, моков, автоподстановок из env или localStorage для username.
7. После регистрации стримера и подтверждения его TikTok username система сама должна отслеживать его эфиры без ручной кнопки подключиться к каждому новому эфиру.
8. Новый frontend и новое ТЗ сразу проектируем под RU и EN, даже если на первом этапе часть контента визуально будет собрана в RU-first режиме.

## Две роли пользователей

### 1. Зритель

Зритель:

- регистрируется отдельно от стримера
- указывает свой TikTok username вручную
- получает профиль внутри платформы
- может подписываться на стримеров
- может получать задания, очки, уровни, рейтинг, награды
- может участвовать в бустах, рейдах и акциях
- может вводить кодовые слова
- может получать уведомления в web и Telegram

### 2. Стример

Стример:

- регистрируется через отдельный entry point, например отдельную кнопку Ты стример
- указывает TikTok username вручную
- может дополнительно пройти подтверждение владения аккаунтом
- получает собственную страницу внутри платформы
- может публиковать контент
- может запускать буст и запрашивать помощь
- может настраивать внешний вид своей страницы
- может подключать Telegram-канал и Telegram-инструменты
- в будущем может запускать платные активности, подписки и промо-механики

## Регистрация и онбординг

### Для зрителя

Минимум:

- email или удобная аутентификация
- passwordless или OAuth можно рассмотреть позже
- обязательное поле TikTok username
- выбор любимого стримера или привязка по рефералке

### Для стримера

Минимум:

- отдельная регистрация
- обязательный TikTok username
- профиль стримера
- возможность подать заявку на отслеживание аккаунта

### Подтверждение TikTok аккаунта

Варианты подтверждения:

1. Простая версия MVP: стример вручную вводит TikTok username, а админ подтверждает аккаунт.
2. Улучшенная версия: стример выполняет действие для верификации, например размещает код в bio или делает контрольное действие.
3. Позже можно добавить OAuth, если TikTok действительно даст это сделать и это будет практично для продукта.

На текущем этапе нельзя строить продукт вокруг надежды на полноценный TikTok OAuth. Архитектура должна работать и без него.

## Что обязательно должно быть в backend

Backend - это ядро NovaBoost Live.

Он должен отвечать за:

- отслеживание live-статуса стримеров
- автоматическое подключение трекеров к стримам зарегистрированных стримеров
- сбор событий стрима
- сбор и нормализацию событий зрителей
- начисление очков и бонусов
- обработку бустов, рейдов и сигналов помощи
- подписки и уведомления
- Telegram-бота и Telegram-интеграции
- алгоритм распределения трафика
- антиспам и ограничение шума
- аудит событий и аналитический контур

## Принцип отслеживания стримов

Система должна работать так:

1. Стример зарегистрирован в платформе и указал TikTok username вручную.
2. Backend добавляет этого стримера в пул отслеживания.
3. Сервис мониторинга проверяет live-статус стримера.
4. Когда стример выходит в эфир, система автоматически поднимает worker или tracker-session.
5. Worker подключается к источнику live-событий через выбранную библиотеку или адаптер.
6. Дальше backend получает события стрима: старт, стоп, viewers, likes, gifts, chat, joins и другие доступные сигналы.
7. Эти события пишутся в event pipeline, обрабатываются и используются для UI, очков, уведомлений, рейтингов и алгоритма продвижения.

Важно:

- стример не должен вручную нажимать подключиться к каждому эфиру
- ручной ввод username нужен на этапе регистрации и настройки
- автоматическое отслеживание начинается после добавления стримера в систему

## Источник live-событий

Для TikTok tracking можно использовать адаптерный подход:

- tracker adapter interface
- реализация на базе TikTok Live Connector или другой рабочей библиотеки
- возможность заменить библиотеку без переписывания всей бизнес-логики

Мы не должны завязывать архитектуру всего продукта на одну библиотеку напрямую.

Нужно строить backend вокруг событий.

Базовые типы событий:

- streamer.live_started
- streamer.live_ended
- stream.viewer_joined
- stream.viewer_left
- stream.like_received
- stream.gift_received
- stream.chat_message
- stream.snapshot_updated
- viewer.task_completed
- viewer.code_submitted
- boost.started
- boost.expired
- raid.requested

Каждое событие должно иметь:

- event id
- event type
- timestamp
- source
- streamer id
- stream session id
- viewer id, если применимо
- raw payload
- normalized payload

## Очки, бонусы и участие зрителей

Платформа должна начислять очки не за кнопку ради кнопки, а за реальные действия.

Примеры:

- зашел на эфир и пробыл достаточно времени
- ввел правильное кодовое слово
- участвовал в бусте
- пришел по сигналу нужен буст
- вернулся повторно на эфир стримера
- отправил лайки
- отправил подарки, если событие доступно и это корректно отслеживается
- участвовал в активности платформы или акции

На базе этого строятся:

- points
- level
- activity score
- streaks
- referral bonus
- loyalty bonus

## Алгоритм платформы

Платформа должна сама решать, кого показывать и кому давать трафик.

На первом этапе score можно строить из:

- active boost
- текущая live-активность
- engagement
- retention signal
- referral strength
- ручные флаги платформы

Пример концептуальной формулы:

priority_score = boost_score + engagement_score + referral_score + retention_score + platform_signal

Это не финальная математика, а направление. Формула должна быть настраиваемой и храниться на backend, а не быть зашитой во frontend.

## Telegram как часть системы

Telegram - это не внешний бонус, а отдельный интерфейс продукта.

### Telegram-бот платформы должен уметь:

- отправлять уведомления о старте эфира
- отправлять сигнал нужен буст
- принимать кодовые слова
- запускать простые механики участия
- показывать ссылки на эфир и страницу стримера
- в будущем давать задания, статусы, очки и подписки

### Интеграция стримера с Telegram

Стример может:

- подключить Telegram-канал или Telegram-аккаунт
- использовать платформу для публикации анонсов и контента
- получать инструменты удержания и вовлечения аудитории

## Контентный слой и мини-соцсеть

У стримера внутри платформы должен быть контентный слой:

- посты
- новости
- анонсы
- уведомления для подписчиков
- короткие видео или ссылки на TikTok-видео
- закрепленные блоки
- кастомные секции

Это нужно, чтобы платформа жила не только во время прямого эфира.

## Предлагаемая архитектура backend

### Основные сервисы

1. API service
- auth
- profiles
- streamers
- viewer actions
- subscriptions
- content
- boosts
- tasks
- referrals

2. Stream tracking service
- мониторит live-статусы стримеров
- запускает tracking workers
- завершает tracking sessions

3. Stream worker
- подключается к live-источнику конкретного стримера
- получает raw events
- нормализует события
- отправляет их в event bus

4. Event processing service
- считает очки
- обновляет агрегаты
- обновляет stats и rankings
- подготавливает триггеры для уведомлений

5. Notification service
- web notifications
- Telegram notifications
- антиспам
- rate limit

6. Telegram bot service
- bot commands
- кодовые слова
- deep links
- уведомления

7. Scoring and traffic engine
- считает priority score
- решает, кому давать внимание и показы
- управляет логикой нужен буст, promoted, top, raid

### Инфраструктура

- Node.js backend
- PostgreSQL как основная база
- Redis для очередей, кэша, rate limit и короткоживущих live-state данных
- WebSocket или SSE для real-time доставки на frontend
- workers для стрим-трекинга и event processing

## Real-time контур

Frontend не должен напрямую определять бизнес-логику жизни платформы.

Frontend получает с backend:

- live stream lists
- promoted lists
- needs boost lists
- streamer page data
- content feed
- notifications
- viewer stats
- leaderboard data

Frontend отправляет в backend:

- подписка на стримера
- попытка выполнить задание
- запуск буста
- ввод кода
- действия пользователя

## Что нельзя делать

- нельзя строить core-логику только во frontend
- нельзя завязывать систему на ручное переключение стримером каждого эфира
- нельзя считать fake-completion заданий без реального подтверждения действий
- нельзя делать TikTok username через автоподстановку по умолчанию
- нельзя оставлять Telegram как необязательное потом когда-нибудь, если он заявлен как важная часть продукта

## Что делаем прямо сейчас

Текущий приоритет:

1. Зафиксировать продуктовую и техническую модель.
2. Перестроить frontend под реальный продукт, а не только под витрину.
3. Подготовить backend-архитектуру под stream tracking и event processing.
4. После этого поэтапно подключать реальную backend-логику.

## Рабочая формула команды

Мы строим не просто интерфейс и не просто таблицы.

Мы строим живую платформу, где:

- стример получает рост
- зритель получает участие
- backend собирает события и считает ценность
- frontend показывает жизнь системы
- Telegram расширяет удержание и вовлечение

Именно backend является сердцем NovaBoost Live, потому что платформа должна видеть, понимать и направлять движение вокруг эфиров.
